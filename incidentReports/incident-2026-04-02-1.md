## Summary

The JWT Pizza service experienced a complete failure of being able to fufill the pizza orders. I believe that it was from approximatley 11 am until I resolved the problem at 3:20 pm on April 2. The cause was due to injected choas event that made it so then the pizza factory API key was invaild. This made it so then all POST /api/order requests that made it so then the request would return the error "Failed to fulfill order at factory." The rror was detected by the Granafa dashboard, which both showed a spike in active users and pizza creation failures. I started to investigate the Grafana logs, fixing a misconfigured dashboard logging panel (which was a separate issue and not the root cause), and in the end I made a curl request against the order endpoint which showed a `followLinkToEndChaos' URL that was inside the error response. Hitting that URL fixed the chaos. This high severity incident affected 100% of users attempting to place pizza orders during the window.

There was further impact as noted by an initial misdiagnosis at 11:30am that delayed full resolution by approximately 2 hours, while I first thought that the chaotic statues was fixed but after later investigation after my classes I was able to find the true fix to the problem and resolve the issue.

## Response

The on-call engineer (Chloe Warner) self-detected the incident at approximately 11:30 MT upon returning from an away period. Investigation began immediately in Grafana logs and metrics.

At 12:00am MT, a separate dashboard logging misconfiguration was identified and fixed, which was mistakenly believed to be the root cause. The engineer stepped away for class at approximately 12:30 MT without confirming end-to-end order success, delaying full resolution by approximately 2 hours.

Upon returning at 3:15pm MT, the engineer resumed investigation and identified the true root cause via a curl test at 3:20 MT.

## Root cause

The instructor chaos system invalidated the Pizza Factory API key. The JWT Pizza backend uses this key to authenticate all order fulfillment requests to the Pizza Factory. With the key invalidated, every POST /api/order request failed with a 500 error. The resolution URL was inside in the response from the start but was not inspected until late in the investigation.

## Resolution

The incident was resolved by calling the followLinkToEndChaos URL that was insdie in the factory error response. 
IT was fixed and confirmed by placing a test order using a curl whihc returned the ID with valid pizza JWT.

To improve time to resolution in future incidents, the full error  should be inspected immediately, and there should have been more tests to make sure that the fix was the correct actions that were needed to be taken. Along with this, making a few more notifications so then it the error continues there are more than one alert so then once the first one is resolved if the issue continues the error will continue to alert.


## Prevention

This was the first chaos injection incident on this system. No prior incidents with the same root cause was implimented at that level. Doing a few more testing chaos would have been benfical, along with having curl sciptres runniong that would order pizza instead of just logging in and out would have been benefical for this specfic example. 

To prevent recurrence, the traffic simulator with pizza orders should always be running during the chaos testing window so that order failures immediately surface in metrics and trigger alerts.
